1. General Questions about QA and Role

What is the role of QA in software development?

The role of QA (Quality Assurance) is not only about finding defects but also about preventing them and ensuring that the final product meets all requirements and user expectations. QA contributes at all stages of the Software Development Life Cycle (SDLC): from requirements analysis to release and maintenance.

In a Sprint: My role involves analyzing requirements, estimating test coverage, designing, creating, and executing test cases/checklists, performing regression testing, testing new features, reporting defects, and close collaboration with developers and the Product Manager.

As a QA, how do you see your role and tasks in the sprint (Agile/Scrum)?

See answer above - the role includes requirements analysis, test coverage estimation, test case design, test execution, regression, and team collaboration.

2. Testing: Basics and Classification

What is the classification of testing types?

Classification can be:

  • By purpose: Functional, Non-functional
  • By level: Unit, Integration, System, Acceptance
  • By positive/negative scenarios
  • By execution: Static/Dynamic
  • By structure: Black/White/Gray box
What levels of testing are there?

UnitTests individual components/functions
IntegrationTests the interaction between modules
SystemTests the system as a whole
AcceptanceConfirms readiness for the end-user/client

What is regression testing, and when do we use it?

Regression Testing: Retesting previously working functions after code changes (bug fixes, new features). Used to ensure changes haven't broken existing functionality.

Do we use regression testing for each feature?

No. Regression is typically run after a feature is complete and merged to check that it hasn't impacted other parts of the system. For the feature itself, we use functional testing.

What types of testing should we use before release?
  • Regression Testing (full or partial)
  • Smoke Testing
  • Sanity Testing
  • User Acceptance Testing (UAT)
  • Non-functional testing (Performance, Security)

3. Testing: Methods and Design Techniques

What are some common testing techniques you use?
  • Equivalence Partitioning
  • Boundary Value Analysis
  • Decision Tables
  • State Transition Testing
  • Exploratory Testing
Can you explain the difference between black-box and white-box testing?

‚¨õ Black Box

  • Testing functionality without knowledge of internal code structure
  • Focus on requirements and user behavior

‚¨ú White Box

  • Testing with access to source code
  • Focus on code coverage (statements, branches, conditions)
White box testing techniques
  • Statement Coverage
  • Branch/Decision Coverage
  • Condition Coverage
  • Path Coverage
Examples of grey box testing (e.g., API or DB)

Gray Box: Testing with partial knowledge of the internal structure.

Example: API testing (knowing request/response structure and perhaps DB structure) or testing that requires database interaction.

What do we achieve using boundary values and equivalence partitioning?

Equivalence Partitioning: Reduces the number of test cases by selecting one representative value from each class (valid and invalid).

Boundary Values: Finds errors at the edges of a range (maximum, minimum values).

Goal: Reduce test redundancy and increase defect-finding efficiency.

Decision Tables - when is it better to use them and what for?

Used when system behavior depends on a combination of several conditions (rules). Useful for complex business logic.

Pairwise testing - where could it be useful?

Used for combinatorial testing where all possible pairs of input parameters need to be checked. Useful for testing complex forms, OS/browser configurations, where full coverage is impractical, but the chance of an error from two parameters interacting is high.

How to understand that the number of test cases is enough?

Determined by the level of Test Coverage:

  • Requirements coverage
  • Code coverage (for white box)
  • Passing of all test cases without finding critical/blocking defects

Principle: We can never prove the absence of defects, but we can ensure we have covered all risks.

What is an example of a non-functional bug?

Non-functional Bug: A defect related not to what the system does, but to how it does it.

Examples:

  • Page load takes more than 10 seconds (Performance issue)
  • Password is shown in plain text in the logs (Security issue)
What is localization testing? How to perform it when changing languages?

Localization Testing: Verifying that the product is adapted for a specific language, region, and culture.

When changing languages, test:

  • Translation (correctness, context)
  • Date/time/currency formats
  • Correct text display (e.g., RTL for Arabic)
  • No "cut-off" strings (i18n)
What is accessibility testing?

Accessibility Testing: Verifying that the product can be used by people with disabilities (vision, hearing, motor skills).

Checks include:

  • Keyboard navigation
  • Screen reader support
  • Color contrast (WCAG)

4. Documentation and Test Management

Test documentation - what is STP, STD, STR?

According to IEEE 829:

  • STP (Software Test Plan): Document describing the scope, approach, resources, and schedule of planned testing activities
  • STD (Software Test Design): Description of how to perform a specific test
  • STR (Software Test Report): Document containing the summary of testing activities and an evaluation of the quality of the tested item
What is a Test Plan? Who is responsible? Who should have access?

Test Plan: A high-level document outlining objectives, scope, approach, schedule, entry/exit criteria, environment, and responsible parties for testing.

Responsibility: QA Lead/Test Manager or an experienced QA Engineer

Access: The entire development team (Developers, Product Managers, QA, Management) to understand the process and assess risks

What is Test Strategy? Difference between Test Plan and Test Strategy?

Test Strategy: A higher-level document (often part of a general project/company document) that defines the general methods and approaches to testing used within the organization or for a group of projects.

Difference:

  • Strategy: What and How we will test (the general vision)
  • Plan: When, Who, Where, and How Much (a specific plan of action for a project)
What is the difference between test case and checklist?

Test Case

Formal document with clear steps, expected result, and pass/fail criteria. Used when high detail and stability are required (e.g., regression, critical functionality).

Checklist

A list of items or points to check without detailed steps. Quick and flexible. Used for exploratory testing or when requirements are unclear/changing.

Checklist - when is it better to use them and when is it possible to manage only with them?

When to use:

  • For exploratory testing
  • When requirements are unclear/constantly changing
  • For testing simple, non-critical functionality/UI

A project can survive with only a checklist (especially in fast-paced Agile), but it increases risks as results are less reproducible and coverage is not clearly documented.

Describe how a good test case should look like

A good test case should be:

  • Independent - executable alone
  • Accurate - clear steps and expected result
  • Economical - not redundant
  • Reproducible - gives the same result every time
  • Have a Unique ID/Title
  • Be linked to a requirement
Have you used any test management tools? If so, which ones?

Tools: Jira, Confluence, TestRail, Zephyr, Qase, Xray

Who needs a test plan?
  • QA Lead/Manager: to organize work
  • Developers: to understand the testing scope
  • Product Managers/Clients: to understand risks and evaluate quality/timelines

5. Defect Management and Prioritization

How do you prioritize testing tasks when faced with limited time?

Risk-Based Testing is applied.

First, test:

  • Critical functionality (main revenue/user path)
  • Areas with high risk (heavily modified code, complex integration)
  • Smoke tests and key regression tests
Severity/Priority - difference, examples

üî¥ Severity

  • Reflects the impact of the defect on the system/functionality
  • Determined by the tester

Example: Blocker - the system is unusable

üü† Priority

  • Reflects the urgency of fixing the defect
  • Determined by the Product Manager/QA Lead

Example: High - must be fixed in the current sprint

On what should a tester pay more attention - on severity or on priority?

The tester should focus on Severity, as it is their expert assessment of the bug's impact.

However, Priority determines when the bug will be fixed, so in the workflow, Priority is more important (as agreed upon with the team).

How do you handle communication with developers when reporting bugs?

Professionalism: Provide a clear, reproducible bug report (Steps, Expected/Actual Result, Environment, Logs/Screenshots/Video).

Collaboration: Discuss the bug, don't argue. Ask if any additional information is needed. Avoid accusatory language.

Reproduction: Offer to reproduce the bug together if the developer cannot see it.

Can you describe a challenging bug you've encountered and how you resolved it?
‚ö†Ô∏è This should be your personal example!

Structure: Problem ‚Üí Cause ‚Üí How Found/Resolved

Example: The bug appeared only in a specific combination of OS/Browser/Screen Resolution. Resolution: Using DevTools and proxy tools (Charles/Fiddler), I found that a specific API request header was incorrectly sent in that environment due to a peculiar browser setting.

What actions can be done if a developer does not agree that there is a bug?
  1. Re-verify: Repeat the steps together on the developer's machine or a shared test environment
  2. Check Requirements: Refer to the original Requirements/Specification (User Story, Acceptance Criteria) to confirm that the current behavior does not match the expected behavior
  3. Escalate to PM/Lead: If agreement is not reached, involve the Product Manager or QA Lead/Tech Lead to make the final decision on whether it is a bug or a change in requirements

6. Web, Cross-browser, and Mobile Testing

How do you approach testing for different platforms (e.g., web, mobile)?

üåê Web

  • Focus on cross-browser compatibility
  • Responsive design
  • Page load performance

üì± Mobile

  • Focus on screen orientation
  • Interruptions (calls/notifications)
  • Gesture usage
  • Battery/data consumption
  • OS specifics (iOS/Android)
Cache/cookies - difference, usage

Cache

Stores static files (images, CSS, JS) to speed up site loading on repeat visits.

Goal: Speed

Cookies

Small text files that store user data (sessions, preferences, login). Sent with every server request.

Goal: Session state/Personalization

Cross-browser testing - how to perform?

What it is: Verification that the web application works consistently (or acceptably) across all supported browsers/OS.

Functional vs. UI:

  • Functional testing (e.g., login works) can often be done in one browser, as the server-side logic or the browser's JS engine usually handles it similarly
  • UI/Visual testing must be performed in several because rendering engines (Blink, Gecko, WebKit) display CSS and HTML differently
What is the best way to perform 100 test cases for 4 browsers?

Automation. The best way is to create automated tests that can be run in parallel on different browsers (Selenium Grid, Cypress, Playwright) or use cloud platforms (BrowserStack, Sauce Labs).

Manually execute only the most critical, non-automatable, or visual tests.

Is it better to use real devices or emulators?

Emulators/Simulators

  • + Fast, convenient for development/early testing
  • ‚àí Don't mimic real-world conditions (battery, network, interruptions)

Real Devices

  • Essential for final testing
  • Accurately simulate hardware limitations
  • Interruptions, network operation, and performance

Conclusion: Use emulators for most functional tests, and real devices for non-functional and critical acceptance tests.

What specific can we name about mobile testing?

Mobile Testing Specifics:

  • Interruptions: Incoming calls, notifications, low battery
  • Performance: Battery and RAM consumption
  • Network: Switching between Wi-Fi/LTE, loss of connection
  • Gestures: Swipes, pinch-to-zoom, long presses
  • Orientation: Switching between portrait/landscape modes

7. Requirements Analysis

How do we perform requirements analysis ("grey zones" and so on)?

Performing Requirements Analysis:

  • Studying all provided documentation (User Stories, Specifications, Mockups)
  • The main task of QA is to identify "grey zones" (uncertainties, omissions) by asking "What if..." questions
  • Breaking down requirements into atomic (smallest) parts (Acceptance Criteria, BDD Scenarios) to create test cases
What could be the typical problems while requirements analysis?

Typical Problems:

  • Incompleteness: A requirement does not describe what should happen in a negative scenario (e.g., entering invalid data)
  • Contradictory: Two requirements conflict with each other
  • Ambiguity: A requirement can be interpreted in different ways (e.g., "The system should be fast")
  • Untestability: The requirement does not contain criteria by which its fulfillment can be evaluated
How do you perform requirements analysis (breaking down to the smallest features)?

Breaking Down to Features: Using techniques such as Mind Mapping or User Story Mapping to visualize and decompose requirements.

Each requirement must be converted into a set of Acceptance Criteria, which serve as the basis for writing test cases.

8. API Testing

What is API - methods, codes?

API (Application Programming Interface): A set of protocols, routines, and tools for building software, allowing two different software components to communicate.

Postman - variables, dependencies

Postman (Variables): Used to store data that may change or repeat (e.g., base URL, authorization token).

Storing the URL in Environment Variables or Global Variables avoids re-entering it.

Dependencies: (e.g., fetching an authorization token) are handled via Pre-request Scripts (JS code).

What HTTP(S) methods are and what are they?
MethodCRUDDescription
GETReadRetrieve data. Idempotent and safe
POSTCreateCreate new entity. Not idempotent
PUTUpdateComplete update. Idempotent
DELETEDeleteDelete entity. Idempotent
PATCHUpdatePartial update
Example of structure of the GET request/response

Start Line: GET /users/123 HTTP/1.1

Headers: Host: api.example.com, Accept: application/json

Query Parameters: /users?status=active (part of the URL)

Body: Should not contain a body (according to best practices)

What is the difference between PUT and POST methods?

POST: Creates a new resource and is not idempotent.

PUT: Replaces/updates a resource entirely and is idempotent (executing multiple times yields the same result).

Types of response codes and meaning
  • 1xx (Informational): Information (e.g., 100 Continue)
  • 2xx (Success): Success (e.g., 200 OK, 201 Created, 204 No Content)
  • 3xx (Redirection): Redirection (e.g., 301 Moved Permanently, 302 Found)
  • 4xx (Client Error): Client Error (e.g., 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found)
  • 5xx (Server Error): Server Error (e.g., 500 Internal Server Error, 503 Service Unavailable)
Can GET request contain body request?

No. A GET request should not contain a body (according to best practices).

When should we use API testing?
  • Before UI testing
  • For testing business logic (independently of UI)
  • For testing integration between services
  • For security verification (authorization, data validation)
What are the most common types of bugs we can meet in API testing?
  • Incorrect response code (e.g., 200 instead of 404)
  • Incorrect input data validation (e.g., accepts an empty mandatory parameter)
  • Security issues (missing authorization/authentication)
  • Performance issues (slow response)
  • Incorrect response format (JSON/XML)
What is the structure of the POST request (header, body request, parameters)?

Start Line: POST /users HTTP/1.1

Headers: Content-Type: application/json

Body: Contains data to create the resource (JSON/XML)

9. Databases (SQL)

SQL - what is it, what is used for?

SQL (Structured Query Language): A standardized language for managing data in relational database management systems (RDBMS). Used for creating, reading, updating, and deleting (CRUD) data.

What types of SQL commands do you know?
TypeDescriptionCommands
DDLData Definition Language - DB structureCREATE, ALTER, DROP
DMLData Manipulation Language - dataSELECT, INSERT, UPDATE, DELETE
DCLData Control Language - access controlGRANT, REVOKE
TCLTransaction Control Language - transactionsCOMMIT, ROLLBACK
Commands, functions and operators in SQL

Commands (Statements): SELECT, INSERT, UPDATE, DELETE, CREATE TABLE

Functions: Perform calculations on data (COUNT(), SUM(), AVG(), MAX(), MIN(), UPPER(), CONCAT())

Operators: Used in conditions (=, >, <, AND, OR, LIKE, IN)

Difference between database and datatable

Database: A container that holds all objects (tables, views, indexes, etc.)

Datatable (Table): An object within the DB that stores data in a structured format (rows and columns)

Examples of functions in SELECT request

Aggregate functions: COUNT(*), SUM(column_name), AVG(column_name)

Scalar functions: UPPER(), CONCAT()

What types of JOIN there are? What is the difference between LEFT JOIN and RIGHT JOIN?
  • INNER JOIN: Returns rows that have matching values in both tables
  • LEFT (OUTER) JOIN: Returns all rows from the left table and matching rows from the right. If there is no match, the right side will be filled with NULL
  • RIGHT (OUTER) JOIN: Returns all rows from the right table and matching rows from the left. If there is no match, the left side will be filled with NULL
  • FULL (OUTER) JOIN: Returns rows when there is a match in one of the tables
When working with database could be useful for QA tester?
  • Data Verification: Checking that data sent via UI/API is correctly saved in the DB
  • Test Data Setup: Creating or modifying data for testing specific scenarios
  • Business Logic Testing: Verifying complex logic implemented at the DB level (stored procedures, triggers)
  • Diagnosis: Assisting in determining the cause of a bug (e.g., finding a discrepancy between the UI and DB)

10. Tools and Methodologies (Agile, DevTools, Git, JS)

What is Agile/Scrum?

Agile: A software development philosophy based on incremental delivery, collaboration, and rapid response to change.

Scrum: A framework for implementing Agile, using iterations (sprints).

Roles: Scrum Master, Product Owner, Development Team

Events: Sprint Planning, Daily Scrum (Stand-up), Sprint Review, Sprint Retrospective

Automated test - area of usage, when should we apply automated testing?

When to apply:

  • Tests executed repeatedly
  • Stable functionality
  • Regression testing
  • Cross-browser testing
  • Load testing

When NOT to apply:

  • Very small project
  • Short lifecycle
  • UI changes too frequently (high maintenance cost)
What types of selectors can we use in UI automated testing?

ID, Name, Class Name, Tag Name, XPath, CSS Selectors (most reliable)

What is git and github? How can we send changes of the code to the main repo from the local repo?

Git: A distributed version control system for tracking code changes.

GitHub: A cloud-based service for hosting Git repositories.

Send to main repo: After git add . and git commit -m "Message", use git push origin <branch-name>. Then create a Pull Request (PR) to merge into main or master.

Merge changes: git merge <branch-name> (merges the specified branch into the current one) or use a PR on GitHub.

Resolve conflicts: Manually edit the conflicting files, choose the desired changes, then git add <conflicted-file> and git commit.

What is Charles is used for?

A Proxy Server for recording and analyzing all traffic between a web browser (or mobile application) and the Internet. Used to view/modify requests/responses (e.g., for testing 500/404 errors).

How can we override responses in DevTools and for what?

Allows intercepting and modifying HTTP responses/files without changing server code.

Used for:

  • Simulating a slow/failed API response (e.g., 500/404 code)
  • Testing UI edge cases with a large amount of data
What is the most useful feature of DevTools for a QA tester?

Console: for viewing JS/API logs and errors

Network: for analyzing all HTTP requests and responses, including load time, headers, and body

What are the main data types in JavaScript? How to create function? What loops are in JS?

Data Types:

  • Primitive: string, number, boolean, null, undefined, symbol, bigint
  • Non-Primitive: object (including Array, Function)

Create Function: function functionName(params) { ... } or as an arrow function const functionName = (params) => { ... }

Loops: for, while, do...while, for...in (for object properties), for...of (for iterable objects like arrays)
